# Repository Guidelines

## Project Structure & Modules
- `src/` TypeScript source: `index.ts` bootstraps the MCP server and registers tools; `tools/` contains ClickUp tool handlers (task, list, time, document, etc.); `shared/` holds config/utilities (e.g., `shared/config.ts`, `shared/utils.ts`); `resources/` stores static space resource helpers.
- `src/tests/` Node test files (`*.test.ts`) covering task, document, and time flows; run with the built-in Node test runner.
- `dist/` compiled JS + type declarations generated by `npm run build`; never edit directly.
- Root configs: `tsconfig.json`, `manifest.json`, `docker-compose.yml`, `Dockerfile`, `.env.example`.

## Build, Test, and Development Commands
- `npm install` ‚Äî install dependencies (Node ‚â•16).
- `npm run build` ‚Äî compile TypeScript to `dist/` via `tsc`.
- `npm start` ‚Äî run the compiled MCP server from `dist/index.js`.
- `npm run dev` ‚Äî watch/auto-compile with `tsc -w` and restart via `nodemon dist/index.js` (good for iterative work).
- `npm run cli` ‚Äî execute the CLI entrypoint directly with `ts-node src/cli.ts` (helpful for quick manual calls).
- `npm run test` ‚Äî run Node‚Äôs `--test` runner with `ts-node/register` over `src/**/*.test.ts`.
- `npm run prettier` ‚Äî format `src/**/*.ts` using Prettier defaults.
- Packaging: `npm run mcpb` to build and create `ClickUp.mcpb`; `npm run release` to build, publish to npm, tag, and push.

## Coding Style & Naming Conventions
- Language: TypeScript targeting ES2020, `strict` mode enabled (see `tsconfig.json`); prefer async/await and typed responses.
- Formatting: Prettier defaults; 2-space indentation; import order kept logical (external ‚Üí internal).
- Naming: files in `kebab-case.ts`; exported types/interfaces in `PascalCase`; functions in `camelCase`; tests mirror source concepts with `.test.ts` suffix under `src/tests/`.
- Error handling: propagate ClickUp API errors with context; log via `console.error` where initialization matters (e.g., server startup).

## Testing Guidelines
- Test runner: Node `--test` with TypeScript via `ts-node/register`; no Jest-specific globals.
- Add new tests in `src/tests/`, naming the file after the feature (e.g., `createTask.test.ts`); prefer `describe`/`it` blocks that cover success and failure (auth/mode checks).
- Integration tests should stub external HTTP calls; avoid hitting live ClickUp APIs‚Äîinject fixtures or mock HTTP clients.
- Run `npm run test` before pushes; keep tests independent of local `.env` secrets when possible.

## Commit & Pull Request Guidelines
- Commits: short, imperative summaries mirroring existing history (e.g., ‚ÄúAdd deleteTask tool‚Äù, ‚ÄúRelease v1.6.0‚Äù); keep scope focused.
- PRs: include summary of changes, motivation/linked issue, test results (`npm run test` output), and screenshots of CLI/log output when behavior changes. Note required env vars (`CLICKUP_API_KEY`, `CLICKUP_TEAM_ID`, optional `CLICKUP_MCP_MODE`) for reviewers.

## Security & Configuration Tips
- Copy `.env.example` to `.env`; never commit secrets. Minimum required variables: `CLICKUP_API_KEY` and `CLICKUP_TEAM_ID`; set `CLICKUP_MCP_MODE` to `read-minimal` for safest default during development.
- Transport: `CLICKUP_MCP_TRANSPORT` defaults to `stdio`; set to `http` (plus `PORT`, default 8417) for Docker/network deployments. In HTTP mode, API credentials arrive via headers `X-ClickUp-API-Key` and `X-ClickUp-Team-ID`; do not bake them into images.
- Docker: `docker-compose up --build` maps `8417:8417` with `CLICKUP_MCP_TRANSPORT=http`; adjust `CLICKUP_MCP_MODE` via env. Ensure `dist/` is built before building the image.
- When adding tools under `src/tools/`, respect mode gating (read/read-minimal/write) and keep descriptions clear for downstream AI agents.
